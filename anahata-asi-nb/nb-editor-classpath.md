The goal is to be able to use the NbEditorUI with an in memory or tmp dir based FileObject for activating syntax highlighting on snippets such as 

Markup codeblcks in the chat (global classpath)
compileAndExecute (classpath from the java tool + extraClasspath)
compileAndExecuteInProject (classpath from the java tool + calculated extra classpath from NbJava (project and possibly project deps) + extraClasspath)
createTextFile (project classpath where the file will be created)

As well as normal java project files (for the context panel resource viewer) - less priority but also needed

### Discoveries & Solutions

1. **The Classpath Loss Issue**:
   When opening an in-memory `FileObject` (e.g., `mem:///demo/DemoSnippet.java`), the `JavacParser` attempts to resolve its classpath. It delegates to `ClassPath.getClassPath(fo, type)`, which queries all `ClassPathProvider`s in the global `Lookup`. Since the memory file doesn't belong to a NetBeans Project, all providers return `null`. The parser falls back to an empty classpath, preventing `SemanticHighlighter` from resolving symbols like `List` or `JButton`.

2. **The `boot.jar` Error**:
   When manually constructing a `ClassPath` using `ClassPathSupport.createClassPath(URL[])`, standard file URLs (e.g., `file:/path/to/my.jar`) are rejected for archives. NetBeans requires the `jar:` protocol. We must wrap `File` objects with `FileUtil.urlForArchiveOrDir(file)` to generate compliant URLs.
   **Why it's not the same as bootstrap libraries:** `boot.jar` from the NetBeans toolkit provides the NetBeans module system bootstrapper. It is part of the application's runtime but *not* part of the JDK's standard library. `JavaPlatformManager.getDefault().getDefaultPlatform().getBootstrapLibraries()` specifically targets the JDK classes (`java.lang.*`, etc.).

3. **The JEditorPane `setText()` Trap**:
   Calling `editor.setText(code)` completely discards the existing `Document` and instantiates a new one. If we inject metadata (like `StreamDescriptionProperty` or `JavaSource.class`) *before* setting the text, the properties are lost on the discarded document. The correct sequence is:
   1. `Document doc = editor.getDocument();`
   2. `doc.remove(0, doc.getLength());`
   3. `doc.insertString(0, newText, null);`
   This preserves the original `NbEditorDocument` and its crucial `DataObject` bindings. This was the root cause for **normal project files** failing to show highlighting in the Context Panel!

4. **PathHandle Resolution Bug**:
   `NetBeansTextResourceViewer` was completely ignoring `PathHandle`, failing to resolve a `FileObject` for physical files generated by the `createTextFile` tool. This caused them to fall back to `text/plain` with no `DataObject` binding, which explains why the Popup Demo failed for project files.

5. **Dynamic Global Providers via SystemFileSystem (The Elegant Fix)**:
   NetBeans allows dynamic or declarative registration of `ClassPathProvider`s. We created `AnahataClasspathProvider` with `@ServiceProvider(service = ClassPathProvider.class, position = 10000)`. It detects if a `FileObject` comes from the `MemoryFileSystem` and responds with the `NbJava.getDefaultClasspath()`. This seamlessly integrates with the background parser without any reflection or document-property hacks.
   *Note 1:* `org.openide.filesystems.MemoryFileSystem` is package-private, so we must use `.getClass().getSimpleName().equals("MemoryFileSystem")` instead of `instanceof` to avoid compilation errors.
   *Note 2:* Position 10000 guarantees our provider acts as a fallback *after* standard project providers. This prevents it from overriding valid project file classpath resolutions. Logging inside `findClassPath` must be strictly avoided to prevent flooding the IDE's parsing loop.

6. **The Boot Classpath and JavaPlatformManager**:
   If we return `null` for `ClassPath.BOOT` in our custom `ClassPathProvider`, `JavaSource` automatically falls back to `JavaPlatformManager.getDefault().getDefaultPlatform().getBootstrapLibraries()`. This elegantly solves the issue of providing standard JDK classes (like `java.lang.String`) without manually resolving the boot classpath.

7. **Custom Attributes for ClassPath Context Targeting**:
   Since all virtual resources are created via `StringHandle` which maps to `MemoryFileSystem`, we can set custom attributes on the `FileObject` when it is created (e.g., `fo.setAttribute("anahataClasspathContext", "global")` or `fo.setAttribute("anahataClasspathContext", "/path/to/project")`). Our dynamic `ClassPathProvider` can then read this attribute and decide *which* `ClassPath` to return, enabling distinct classpaths for global chat snippets vs. project-bound `createTextFile` operations!

8. **Missing Error Highlighting (Hints / Annotations)**:
   Semantic highlighting (types, variables) works via `SemanticHighlighter`, but error checking (red squiggles) is usually provided by `HintsTask` / `HintsInvoker` which are executed asynchronously. If errors are not showing up in our embedded viewer, it may be because `HintsTask` relies on a fully realized `CloneableEditor` or checks `OpenProjects` to decide whether to run. We will investigate this next.
